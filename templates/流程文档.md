## 一、完整开发流程

```
┌─────────────────────────────────────────────────────────────────┐
│                     完整开发流程（7个阶段）                        │
└─────────────────────────────────────────────────────────────────┘

阶段1️⃣: 需求讨论
    │
    ├── CEO 与用户确认需求
    ├── 整理需求清单
    └── 用户确认需求方向
    │
    ▼
阶段2️⃣: PRD 编写
    │
    ├── Product Agent 编写 PRD
    ├── 保存到 docs/PRD.md
    ├── CEO 汇报给用户
    └── 用户确认 "确认开发" / "修改XXX"
    │
    ▼
阶段3️⃣: 技术设计
    │
    ├── Developer Agent 输出:
    │   ├── docs/开发文档.md
    │   ├── docs/接口文档.md
    │   └── supabase/migrations/001_init.sql
    ├── CEO 汇报给用户
    └── 用户确认 "确认开发" / "修改XXX"
    │
    ▼
阶段4️⃣: 代码开发
    │
    ├── 创建 GitHub 仓库
    ├── 初始化项目
    ├── 配置 Supabase 数据库
    ├── 编写核心代码
    ├── 自测验证（含 SHA1 验证）
    ├── GitHub 提交
    └── 用户确认预览效果
    │
    ▼
阶段5️⃣: 测试验证（Developer 自测）
    │
    ├── 接口测试（curl/Python）
    ├── 功能测试
    ├── 截图保存
    └── SHA1 验证
    │
    ▼
阶段6️⃣: 交付验收
    │
    ├── CEO 验收
    ├── 用户确认效果
    └── 交付完成
    │
    ▼
阶段7️⃣: 项目收尾
    │
    ├── 清理临时文件
    ├── 更新 MEMORY.md
    └── 归档文档
```

## 二、需求讨论流程

### 2.1 需求讨论话术

```
CEO 向用户确认需求：

"好的！我来帮你完成这个项目开发。

📋 在派发给团队之前，我需要跟你确认几个问题：

🎯 核心目标：
- 这个产品的核心价值是什么？
- 目标用户是谁？
- 解决什么问题？

📱 平台要求：
- 需要支持哪些平台？（iOS/Android/Web）
- 需要哪些第三方集成？

⏰ 优先级：
- 必须有的功能（P0）？
- 最好有的功能（P1）？
- 可以后期加的功能（P2）？

💡 请告诉我你的想法，我来整理成完整需求后再交给团队执行。"
```

### 2.2 需求确认清单

| 检查项 | 内容 | 状态 |
|--------|------|------|
| App 名称 | 确认名称和定位 | □ |
| 核心功能 | P0 至少 3 个 | □ |
| 目标用户 | 明确目标用户群体 | □ |
| 设计风格 | 简约/现代/专业 | □ |
| 技术栈 | Flutter + Supabase | □ |

## 三、PRD 评审流程

### 3.1 Product 输出 PRD

```
产出物: docs/PRD.md

必须包含:
├── 产品概述（背景、目标、用户）
├── 用户故事（至少 3 个）
├── 功能需求（P0/P1/P2 分类）
├── 数据模型（ER 图或 SQL）
├── API 设计
└── 验收标准
```

### 3.2 PRD 确认话术

```
CEO 向用户汇报：

"📋 Product Agent 已完成 PRD！

📁 文档位置：[路径]
📊 内容摘要：
- 用户故事：X 个
- 功能点：X 个
- P0 功能：X 个

⏸️ 请审核 PRD：
- 如需修改，请告诉我
- 如确认通过，请回复"确认开发"，我将继续"
```

## 五、代码开发流程

### 5.1 开发步骤

```
1️⃣ 创建 GitHub 仓库
   gh repo create <项目名> --public --clone

2️⃣ 初始化项目
   ├── Flutter: flutter create <项目名>
   └── 配置 pubspec.yaml

3️⃣ 配置 Supabase
   ├── 创建 supabase/migrations/001_init.sql
   ├── supabase db push
   └── 验证数据库连接

4️⃣ 编写代码
   ├── 遵循 DESIGN-SYSTEM.md
   ├── 使用 components/ 组件库
   └── 遵循 CODING-STANDARDS.md

5️⃣ 自测验证
   ├── flutter analyze (0 errors)
   ├── flutter build ios (成功)
   ├── 启动 App 测试核心功能
   ├── 截图保存到 docs/screenshots/
   └── SHA1 验证
```

### 5.2 自测检查清单

| 检查项 | 要求 | 状态 |
|--------|------|------|
| flutter analyze | 0 errors | □ |
| flutter build ios | 成功 | □ |
| 核心页面可访问 | 全部正常 | □ |
| SHA1 验证 | 截图 SHA1 不同 | □ |
| 深色模式 | 正常显示 | □ |

### 5.3 代码提交规范

```bash
# 提交格式
git add .
git commit -m "feat: 实现核心登录功能"
git push origin dev

# 创建 PR
gh pr create --title "feat: 登录功能" --body "实现用户登录注册功能"
```

## 六、交付验收流程

### 6.1 交付检查清单

| 检查项 | 要求 | 状态 |
|--------|------|------|
| 代码完整性 | 所有功能已实现 | □ |
| 自测通过 | 所有 P0 功能正常 | □ |
| 文档完整 | PRD + 接口文档存在 | □ |

⏸️ 请确认是否交付？

## 八、CEO 铁律

> ⚠️ **CEO 绝对不能做以下事情**

| ❌ CEO 不能做 | ✅ 正确做法 |
|--------------|------------|
| 修改/修复代码 | 派给 Developer |
| 启动 App 测试 | Developer 自测 |
| 直接执行编译命令 | Developer 执行 |
| 修改 PRD/设计文档 | 派给对应 Agent |

## 九、异常处理

### 9.1 Developer 遇到问题

```
1. Developer 遇到技术问题
   → 报告 CEO 描述问题

2. CEO 派给 Developer 修复
   → Developer 修复后重新自测
   → 确认通过后汇报 CEO

3. 阻塞超过 10 分钟
   → Developer 主动汇报
   → CEO 协调资源
```

### 9.2 用户反馈问题

```
1. 用户反馈问题
   → CEO 记录问题

2. CEO 调度 Developer 修复
   → Developer 直接修复

3. Developer 修复后汇报
   → CEO 汇报用户
```

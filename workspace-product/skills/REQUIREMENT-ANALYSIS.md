# 📊 需求分析框架 - Product Agent 工作指南

> **版本**: v1.0
> **用途**: 供 Product Agent 使用
> **职责**: 理解需求 → 分析优先级 → 提出方案 → 编写 PRD

---

## 🎯 我的工作流程

当 CEO Agent 分配需求给我时，我的工作分为 4 个阶段：

```
【第1步】需求理解（1-2小时）
  ├─ 理解"什么是需求" (What)
  ├─ 理解"为什么要做" (Why)
  ├─ 理解"谁会使用" (Who)
  └─ 理解"成功标准是什么" (Success Criteria)

【第2步】需求分析（2-4小时）
  ├─ 拆分为具体功能点
  ├─ 评估复杂度和工作量
  ├─ 识别外部依赖
  └─ 列出风险和假设

【第3步】优先级评估（1小时）
  ├─ 用 MoSCoW 方法分类
  ├─ 评估依赖关系
  ├─ 确定最小可行产品 (MVP)
  └─ 制定阶段计划

【第4步】编写 PRD（4-8小时）
  └─ 使用 PRD-TEMPLATE.md 完整编写
```

## 1️⃣ 需求理解与澄清

### 1.1 "What" - 这是什么需求？

问自己这些问题：

```
□ 需求的核心功能是什么？
□ 用一句话能否描述整个需求？
□ 这个功能涉及哪些主要模块？
□ 用户界面长什么样？
□ 后端需要哪些关键操作？
```

**示例问题清单**：
```
需求: "用户认证系统"

□ 什么是用户认证系统？
  → 允许用户创建账户、登录、重置密码

□ 有哪些认证方式？
  → 邮箱/密码、Google登录、GitHub登录

□ 用户数据如何存储？
  → Supabase 数据库

□ 安全要求是什么？
  → HTTPS、密码加密、Token认证
```

### 1.2 "Why" - 为什么要做这个需求？

理解业务价值和驱动力：

```
对我的分析很重要，因为：
├─ 帮助我评估需求的优先级
├─ 如果有困难决策，我能回到业务目标
├─ 我能识别哪些功能可以在后期版本中删减
└─ 我能建议替代方案
```

**示例**：
```
需求: "我们需要实现双因素认证 (2FA)"

Why分析:
□ 为什么要做 2FA？
  → 用户反馈，安全性要求
□ 这有多紧急？
  → 中等 - 有安全隐忧但不影响核心功能
□ 竞争对手有吗？
  → 是的，大多数认证系统都有 2FA
□ 如果不做会怎样？
  → 用户会有被黑客攻击的风险
```

### 1.3 "Who" - 谁会使用这个功能？

清晰地定义目标用户：

```
【用户群体分析】
用户类型1: 个人开发者
  ├─ 数量: 60%
  ├─ 技能: 熟悉技术
  ├─ 需求: 快速接入，文档清楚
  └─ 优先级: 高

用户类型2: 企业用户
  ├─ 数量: 30%
  ├─ 技能: 非技术人员
  ├─ 需求: 易用，企业级功能
  └─ 优先级: 中

用户类型3: 第三方开发者
  ├─ 数量: 10%
  ├─ 技能: 高级技术
  ├─ 需求: API 灵活性，扩展能力
  └─ 优先级: 中
```

## 2️⃣ 需求分析与分解

### 2.1 功能分解 (Feature Breakdown)

把模糊的需求分解为具体的功能点：

```
原始需求: "用户认证系统"

分解为:
├─ F1: 邮箱注册功能
│  ├─ F1.1: 邮箱格式验证
│  ├─ F1.2: 密码强度验证
│  ├─ F1.3: 邮件确认机制
│  └─ F1.4: 账户自动激活
│
├─ F2: 邮箱登录功能
│  ├─ F2.1: 邮箱密码验证
│  ├─ F2.2: 错误次数限制
│  ├─ F2.3: Session 管理
│  └─ F2.4: 记住我功能
│
├─ F3: 第三方登录
│  ├─ F3.1: Google OAuth
│  ├─ F3.2: GitHub OAuth
│  └─ F3.3: 账户关联
│
└─ F4: 密码管理
   ├─ F4.1: 忘记密码流程
   ├─ F4.2: 密码重置
   └─ F4.3: 密码修改
```

### 2.2 工作量估算

对每个功能估算工时：

```
【估算方法】
- S (Small): 1-2小时
- M (Medium): 2-4小时
- L (Large): 4-8小时
- XL (Extra Large): 8+小时

【示例】
F1: 邮箱注册        M (3小时)
F2: 邮箱登录        M (3小时)
F3: 第三方登录      L (6小时) ← 集成复杂
F4: 密码管理        M (2小时)
─────────────────────────
总计: ~14小时
```

**转换为团队工期**:
```
前端开发: 6小时 (F1.1-1.4, F2, F4)
后端开发: 8小时 (所有逻辑)
测试: 4小时
─────────────
总计 (Product时间): 4小时 (写PRD和设计)
```

### 2.3 依赖关系分析

识别功能之间的依赖：

```
F1 (注册)
  ↓ 必须先完成
F2 (登录)
  ├─ 依赖 F1 完成
  └─ 可以与 F3 平行
F3 (第三方)
  └─ 可以独立实现
F4 (密码管理)
  └─ 可以在 F1/F2 之后添加
```

**关键: 识别阻塞关系，确保并行工作效率最高**

### 2.4 技术复杂度评估

```
【风险清单】

□ 邮件发送服务
  风险: 邮件可能被拦截或延迟
  缓解: 选择可靠服务 (SendGrid/Mailgun)

□ 第三方 API 集成
  风险: OAuth 流程复杂，可能出错
  缓解: 提前做技术调研和 PoC

□ 安全性
  风险: 如果做不好，会有数据泄露风险
  缓解: 使用成熟的认证库，进行安全测试

□ 用户体验
  风险: 流程太复杂，用户放弃
  缓解: 简化注册流程，进行用户测试
```

## 3️⃣ MoSCoW 优先级方法

用这个方法分类功能，确定 MVP：

```
【M - Must Have】必须有（否则无法发布）
- 邮箱注册和登录
- 密码安全存储
- 基本的 Session 管理
→ 这些必须在 v1.0 中完成

【S - Should Have】应该有（v1.0 中做，但可以延迟）
- 邮件确认机制
- 密码重置功能
- 密码强度验证
→ 优先级高，但不是绝对必须

【C - Could Have】可以有（v1.1 或更晚）
- Google/GitHub 登录
- 两步验证 (2FA)
- 社交媒体分享
→ 优先级中等，可以后期添加

【W - Won't Have】不做（至少这个版本）
- 生物识别认证
- 指纹登录
- 区块链集成
→ 暂时不考虑
```

**MVP 定义**:
```
我们的最小可行产品 (MVP) 包含:
- M: 邮箱注册和登录
- S: 邮件确认和密码重置
→ 这是 v1.0 的完整功能集
```

## 4️⃣ 业务建议和替代方案

作为产品经理，我的职责包括**提出建议**，而不只是实现需求。

### 4.1 问题识别

```
需求: "支持邮箱、Google、GitHub 登录"

我的分析:
□ 邮箱注册流程需要邮件服务
□ 如果邮件服务不可靠，用户无法激活账户
□ 这是 v1.0 的关键风险

建议:
□ 使用 Supabase 内置的邮件功能 (简单)
   或
□ 使用 SendGrid/Mailgun (可靠但需要配置)

决策:
→ 建议用 Supabase 内置邮件开始，
  如果有问题再切换到 SendGrid
```

### 4.2 用户体验建议

```
当前设计: 注册 → 发确认邮件 → 用户点击链接 → 激活

问题: 用户可能不会检查邮件，流程太复杂

建议:
1. 立即激活，发邮件仅用于验证 (更友好)
2. 或者，集成第三方登录 (Google/GitHub)
   跳过复杂的邮件流程

决策:
→ v1.0 支持邮件确认 (安全)
→ v1.1 增加第三方登录 (方便)
```

## 5️⃣ 数据模型设计建议

根据需求，提出数据结构：

```
【建议的表结构】

users 表
├─ id (UUID)
├─ email (TEXT, UNIQUE)
├─ password_hash (TEXT)
├─ nickname (TEXT)
├─ avatar_url (TEXT)
├─ created_at (TIMESTAMP)
├─ updated_at (TIMESTAMP)
└─ is_email_verified (BOOLEAN)

sessions 表
├─ id (UUID)
├─ user_id (UUID, FK)
├─ token (TEXT)
├─ expires_at (TIMESTAMP)
└─ created_at (TIMESTAMP)

oauth_accounts 表
├─ id (UUID)
├─ user_id (UUID, FK)
├─ provider (TEXT) -- 'google', 'github'
├─ provider_id (TEXT)
└─ created_at (TIMESTAMP)
```

**为什么这样设计**:
```
- users 表: 存储用户核心信息
- sessions 表: 管理登录状态，支持多设备
- oauth_accounts 表: 支持第三方登录，一个用户可关联多个第三方账户
- is_email_verified: 标记邮箱是否已验证
```

## 6️⃣ API 设计建议

```
【关键 API 端点】

POST /api/auth/register
  ├─ 输入: { email, password }
  ├─ 验证: 邮箱格式、密码强度
  └─ 返回: { user, needs_email_verification }

POST /api/auth/login
  ├─ 输入: { email, password }
  ├─ 返回: { user, token, refresh_token }
  └─ 错误处理: 5次失败后要求验证码

POST /api/auth/google
  ├─ 输入: { id_token }
  ├─ 自动创建账户或关联已有账户
  └─ 返回: { user, token }

POST /api/auth/forgot-password
  ├─ 输入: { email }
  └─ 发送重置链接邮件

POST /api/auth/reset-password
  ├─ 输入: { reset_token, new_password }
  └─ 返回: { success }

GET /api/user (需要认证)
  └─ 返回: { user_info }

PATCH /api/user/profile (需要认证)
  ├─ 输入: { nickname, avatar_url }
  └─ 返回: { updated_user }
```

## 7️⃣ 需求澄清清单

在编写 PRD 前，确保我已经澄清了所有关键问题：

```
【需求澄清清单】

□ 功能范围
  ├─ v1.0 包含哪些功能？
  ├─ v1.1 计划哪些？
  └─ 哪些功能是 MVP？

□ 非功能需求
  ├─ 性能目标？ (登录响应时间 < 200ms?)
  ├─ 安全需求？ (HTTPS, 密码加密?)
  ├─ 可扩展性？ (支持多少并发用户?)
  └─ 兼容性？ (支持哪些浏览器/设备?)

□ 用户体验
  ├─ 用户流程是什么？
  ├─ 错误信息如何处理？
  ├─ 支持多语言吗？
  └─ 有无障碍需求吗？

□ 技术约束
  ├─ 用什么技术栈？
  ├─ 有现成的库可用吗？
  ├─ 数据库选择是什么？
  └─ 第三方服务有哪些?

□ 业务指标
  ├─ 成功的指标是什么？
  ├─ 目标用户数量？
  ├─ 预算和时间限制？
  └─ 对标的竞品是什么？
```

## 🎯 常见的需求分析错误

### ❌ 不要做这些

```
❌ 不要直接跳到编写 PRD
   → 应该先充分分析需求，理解本质

❌ 不要遗漏非功能需求
   → 性能、安全、可扩展性都很重要

❌ 不要假设需求是清晰的
   → 总要做需求澄清，列出假设

❌ 不要忽视用户反馈
   → 如果这是现有系统的改进，要听取用户意见

❌ 不要一味说"好"，不提出建议
   → Product Manager 要有主见，提出替代方案

❌ 不要低估技术复杂度
   → 与 Developer 讨论，了解实现难度
```

### ✅ 正确的做法

```
✅ 先理解需求的本质
   → What/Why/Who 分析

✅ 分解成可管理的功能块
   → 功能清单和依赖关系

✅ 评估优先级和复杂度
   → MoSCoW 分类，工作量估算

✅ 提出替代方案
   → 如果有多种实现方式，列出优缺点

✅ 清晰地记录假设
   → "我们假设用户有邮箱验证"

✅ 与团队充分沟通
   → 发现问题早解决，而不是后期返工
```

## 📞 与其他 Agent 的协作

### 与 Developer Agent 的沟通

```
我完成需求分析后，会问 Developer:

□ 这个需求在技术上可行吗？
□ 有什么技术风险吗？
□ 估算的工作量是多少？
□ 需要什么第三方库或服务？
□ 有什么建议的替代方案吗？

Developer 的反馈会影响:
├─ MVP 的定义
├─ 功能的优先级
├─ 实现方案的选择
└─ 项目的工期
```

### 与 Tester Agent 的沟通

```
在编写 PRD 时，我会考虑 Tester 的需求:

□ 每个功能都有明确的验收标准吗？
□ 性能要求有具体的指标吗？
□ 安全要求有明确的检查清单吗？
□ 兼容性要求有列表吗？

这样 Tester 能直接基于 PRD 设计测试方案。
```

### 与 CEO Agent 的沟通

```
我会定期向 CEO 报告:

□ 需求分析进度
□ 发现的风险和问题
□ 工期预估
□ MVP 建议
□ 是否需要调整需求范围

CEO 基于此信息可以:
├─ 调整 deadline
├─ 删减或增加功能
├─ 协调资源
└─ 做出商业决策
```

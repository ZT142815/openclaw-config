# ✅ 验收标准和完成定义 - Tester Agent 工作指南

> **版本**: v1.0
> **用途**: 供 Tester Agent 使用
> **目标**: 定义"完成"的含义，确保质量标准

---

## 🎯 核心概念

### 什么是 DoD (Definition of Done)?

**DoD** = 一个工作项目必须满足的所有条件，才能被认为是"完成"

```
DoD 不是可选的！
DoD 不因为赶时间就降低！
DoD 是团队的承诺！
```

---

## 1️⃣ Product Agent 的 DoD

Product Agent 的工作（编写 PRD）完成的标准：

### 📋 文档完整性

```
□ PRD 文档存在且完整
  ├─ 项目概览 (Vision/Background)
  ├─ 用户故事 (至少 3 个)
  ├─ 功能列表 (按优先级排列)
  ├─ 数据模型 (ER 图或 SQL)
  ├─ API 设计 (所有端点)
  ├─ UI/UX 说明 (或至少页面流程)
  └─ 上线检查清单

□ 所有用户故事都有验收标准
  └─ 每个故事：As a... I want... So that...
  └─ 每个故事都有 "验收标准" 部分
  └─ 验收标准具体、可测试

□ API 设计完整
  ├─ 所有端点都有详细说明
  ├─ 请求格式明确 (方法、URL、参数)
  ├─ 响应格式明确 (成功和错误情况)
  ├─ 错误码和说明完整
  └─ 有真实的示例

□ 数据模型清晰
  ├─ 所有表都已定义
  ├─ 主键、外键明确
  ├─ 数据类型正确
  ├─ 约束条件完整 (NOT NULL, UNIQUE, 等)
  ├─ 索引已规划
  └─ RLS (行级安全) 已考虑
```

### 🎯 内容质量

```
□ 需求明确，没有歧义
  ├─ 不存在"可能"、"大概"这样的模糊词汇
  ├─ 关键概念有定义
  ├─ 边界情况已考虑
  └─ 假设已列出

□ API 设计遵循规范
  ├─ 使用 RESTful 风格 (或 GraphQL，需一致)
  ├─ HTTP 方法正确 (GET/POST/PUT/DELETE)
  ├─ 状态码正确 (200/201/400/401/404/500)
  └─ 字段命名一致

□ 优先级清晰
  ├─ 用 MoSCoW 或其他方法分类
  ├─ MVP 范围已定义
  ├─ v1.0/v1.1/v1.2 功能已划分
  └─ 依赖关系已标注

□ 与 Developer 对齐
  ├─ API 设计已与 Developer 讨论并获批
  ├─ 技术可行性已确认
  ├─ 工期估算已评估
  └─ 风险已识别
```

### 🔍 评审

```
□ PRD 已被 CEO 审核
  └─ CEO 确认需求清晰、范围合适

□ PRD 已被 Developer 审核
  └─ Developer 确认可实现、工期合理

□ PRD 已被 Tester 审核
  └─ Tester 确认验收标准可测试

□ 没有待解决的评论
  └─ 所有反馈都已处理
```

## 2️⃣ Developer Agent 的 DoD

Developer Agent 的工作（实现功能）完成的标准：

### 💻 代码质量

```
□ 代码已编写
  ├─ 所有功能都已实现
  ├─ 没有 TODO 或 FIXME 注释
  ├─ 没有注释掉的代码
  └─ 没有 debug 代码 (console.log, 等)

□ 代码遵循规范
  ├─ 命名规范一致 (snake_case/camelCase)
  ├─ 代码风格一致 (缩进、空格)
  ├─ 文件结构组织合理
  ├─ 没有魔法数字，常量已定义
  └─ 遵循 SOLID 原则

□ 代码有注释
  ├─ 复杂逻辑有行内注释
  ├─ 函数/类有文档注释
  ├─ API 端点有说明
  └─ 注释清晰、有帮助

□ 没有安全漏洞
  ├─ 没有 SQL 注入风险 (使用参数化查询)
  ├─ 密码已加密 (bcrypt, scrypt, 等)
  ├─ 敏感数据不存储明文 (Token, API Key)
  ├─ 输入已验证和转义
  ├─ 认证/授权检查完整
  └─ CORS 配置正确
```

### 🧪 测试

```
□ 单元测试已编写
  ├─ 覆盖率 >= 80%
  ├─ 所有关键路径都有测试
  ├─ 边界情况有测试
  ├─ 异常情况有测试
  └─ 所有测试都通过

□ 集成测试已编写
  ├─ 主要工作流有集成测试
  └─ 所有集成测试都通过

□ 测试质量高
  ├─ 测试有清晰的描述
  ├─ 测试逻辑清晰
  ├─ 没有测试相互依赖
  └─ 测试运行速度快
```

### 🚀 部署和文档

```
□ 代码已部署到测试环境
  ├─ 部署脚本或说明已提供
  ├─ 环境变量已配置
  ├─ 数据库已初始化
  └─ 测试环境可以访问

□ 文档已更新
  ├─ README 已更新
  ├─ API 文档已生成
  ├─ 部署指南已编写
  ├─ 架构说明已更新
  └─ 更改日志已记录

□ 没有阻塞问题
  ├─ 所有已知的 bug 都已记录
  ├─ 没有"Critical"阻塞问题
  └─ 所有外部依赖都已确认
```

### ✅ 代码审查

```
□ 代码已通过 Code Review
  ├─ 至少 1 位 reviewer 审查过
  ├─ 所有评论都已解决
  ├─ 通过率 >= 90% (选择性忽略 OK)
  └─ 没有待处理的评论

□ Linting 通过
  ├─ 没有 lint 错误
  ├─ 没有警告 (或已确认可忽略)
  └─ 格式化一致
```

## 3️⃣ Tester Agent 的 DoD

Tester Agent 的工作（执行测试）完成的标准：

### 🧪 测试执行

```
□ 测试计划已执行
  ├─ 所有测试用例都运行过
  ├─ 所有 P0 功能都测试过
  ├─ 所有 P1 功能都测试过
  ├─ 所有 P2 功能都测试过 (如果时间允许)
  └─ 所有测试结果已记录

□ 功能测试通过
  ├─ 正常流程全部通过
  ├─ 异常流程全部通过
  ├─ 边界情况全部通过
  └─ 没有 Critical 缺陷未解决

□ 性能测试通过
  ├─ API 响应时间 < 指定值
  ├─ 数据库查询性能正常
  ├─ 没有内存泄漏
  └─ 可以处理预期的并发用户数

□ 兼容性测试完成
  ├─ 支持的浏览器都测试过
  ├─ 支持的设备都测试过
  ├─ 支持的操作系统都测试过
  └─ 没有兼容性问题
```

### 🐛 缺陷管理

```
□ 所有缺陷都已分类和报告
  ├─ 缺陷有清晰的标题
  ├─ 缺陷有详细的步骤来复现
  ├─ 缺陷有期望和实际结果
  ├─ 缺陷有优先级标记 (Critical/Major/Minor)
  └─ 缺陷有附件 (截图、日志、等)

□ Critical 缺陷全部修复
  ├─ 影响核心功能的 bug 都已修复
  ├─ 数据丢失的 bug 都已修复
  ├─ 安全漏洞都已修复
  └─ 修复已验证通过

□ Major 缺陷处理结束
  ├─ 重要功能缺陷都已修复，或
  ├─ 已确认为"拒绝修复"（有充分理由）
  └─ 修复已验证通过

□ Minor 缺陷已处理
  ├─ 已修复的 Minor 缺陷已验证
  ├─ 未修复的 Minor 缺陷已标记为"待后续版本"
  └─ 清单已梳理完成
```

### 📊 测试报告

```
□ 测试报告已生成
  ├─ 测试用例总数和通过数
  ├─ 缺陷统计 (按优先级)
  ├─ 测试覆盖率
  ├─ 与 MVP 需求的对应
  └─ 重点问题总结

□ 报告包含关键信息
  ├─ 测试时间周期
  ├─ 使用的测试工具
  ├─ 测试环境说明
  ├─ 已知的遗留问题
  └─ 发布建议
```

### 🎯 验收确认

```
□ 需求验收标准全部通过
  ├─ 每个用户故事的验收标准都核实过
  ├─ 每个验收标准都通过了测试
  ├─ 没有遗漏的验收标准
  └─ 用户有效负载已验证

□ MVP 功能完整
  ├─ 所有 MVP 功能都可用
  ├─ 没有功能被意外关闭或删除
  ├─ 端到端流程工作正常
  └─ 用户可以完成主要任务

□ 已准备好发布
  ├─ 发布清单所有项都已检查
  ├─ 没有已知的关键问题
  ├─ 性能基准已达到
  ├─ 安全扫描已通过
  └─ 最终签字确认可以发布
```

## 4️⃣ CEO Agent 的 DoD

CEO Agent 的工作（最终验收）完成的标准：

### ✅ 综合验收

```
□ Product 阶段完成
  ├─ PRD 符合所有 DoD 要求
  ├─ API 设计已获 Developer 和 Tester 批准
  └─ 没有待解决的问题

□ Development 阶段完成
  ├─ 代码符合所有 DoD 要求
  ├─ 测试覆盖率 >= 80%
  ├─ Code review 通过率 >= 90%
  └─ 没有待解决的 blocking 问题

□ Testing 阶段完成
  ├─ 测试计划执行完成
  ├─ 没有未解决的 Critical 缺陷
  ├─ 测试报告已生成
  └─ Tester 已签字确认质量

□ 交付物完整
  ├─ 源代码
  ├─ 测试代码和覆盖率报告
  ├─ API 文档
  ├─ 部署指南
  ├─ 测试报告
  ├─ 缺陷清单
  ├─ 变更日志
  └─ 项目总结
```

### 📊 项目指标

```
□ 进度指标
  ├─ 实际工期 vs 计划工期 (差异 < 10%)
  ├─ 各阶段都按计划或更早完成
  └─ 没有意外的长期延迟

□ 质量指标
  ├─ 缺陷率 < 预期
  ├─ Critical 缺陷数 = 0
  ├─ Major 缺陷数 <= 预期数量
  ├─ 测试覆盖率 >= 80%
  └─ 代码审查通过率 >= 90%

□ 成本指标
  ├─ 实际成本 vs 预算 (差异 < 15%)
  ├─ 人工时消耗合理
  └─ 外部服务成本在预期内
```

### 🎉 发布准备

```
□ 发布前检查完成
  ├─ 所有环境变量已配置
  ├─ 数据库已备份
  ├─ 回滚方案已准备
  ├─ 监控和告警已设置
  ├─ 文档已更新并发布
  └─ 团队已培训

□ 沟通准备
  ├─ 用户已通知发布计划
  ├─ 支持团队已了解新功能
  ├─ 关键负责人已确认
  └─ 应急电话已准备

□ 最终确认
  ├─ Product Manager 确认
  ├─ Development Lead 确认
  ├─ QA Lead 确认
  ├─ Operations 确认
  └─ CEO 签字批准 ✅
```

## 6️⃣ DoD 例外处理

### 什么时候可以降低 DoD？

**答：不能！**

但是，在以下情况下，可以用**书面记录**的方式"暂时豁免"某个标准：

```
【允许的例外情况】

1. 时间压力
   例子: "第三方登录功能测试覆盖率只有 65%，
          但 Critical 路径都已覆盖。
          v1.1 会补充剩余的测试。"

   要求:
   ├─ 必须有书面说明
   ├─ 必须明确说明哪些标准被豁免
   ├─ 必须承诺什么时候完成
   ├─ 必须识别风险
   └─ 必须获得 CEO 和相关方签名批准

2. 外部因素
   例子: "Google OAuth API 临时故障，
          无法完成 Google 登录集成测试。
          等待 Google 恢复后立即补测。"

   要求:
   ├─ 必须有充分的理由
   ├─ 不应该定期发生
   ├─ 必须有恢复时间表
   └─ 必须记录在案

【不允许的理由】
❌ "赶时间，这个检查先跳过"
❌ "这个不太重要，可以不做"
❌ "我觉得没问题，不用测试"
❌ "之前项目也这样做过"
```

### DoD 豁免流程

```
1. 识别问题
   → 发现某个 DoD 项无法满足

2. 分析原因
   → 是技术问题、资源问题，还是时间问题？

3. 寻找替代方案
   → 是否有其他方式达到相同的质量？

4. 得到批准
   → 向 CEO 和相关方报告
   → 明确说明风险
   → 获得书面批准

5. 记录豁免
   → 在项目文档中明确记录
   → 承诺补救时间
   → 标记为"已知问题"

6. 后续跟进
   → 按承诺补救
   → 验证补救完成
   → 更新项目记录
```

## 📋 DoD 清单生成

根据项目规模，使用相应的 DoD 清单：

### 小型功能 (1-2 天开发)

```
□ Product: PRD 已编写，API 已设计
□ Developer: 代码已编写，单元测试 >= 80%
□ Tester: 功能测试通过，缺陷已记录
□ CEO: 没有 Critical 缺陷，可以发布
```

### 中型功能 (1-2 周开发)

```
□ Product: 完整 PRD，API 已评审，MVP 已定义
□ Developer: 代码审查通过，集成测试通过，文档已更新
□ Tester: 测试计划已执行，所有 Critical 缺陷已修复
□ CEO: 项目指标符合预期，可以发布
```

### 大型项目 (1 个月以上)

```
□ Product: 完整 PRD，多轮评审，版本规划清晰
□ Developer: Code review >= 90%，性能基准已验证，架构评估已完成
□ Tester: 完整测试报告，性能测试通过，兼容性验证完成
□ CEO: 所有指标符合，发布前检查完成，团队已培训，可以发布
```

## 🎯 关键要点

```
1️⃣ DoD 是团队的承诺，不能妥协
2️⃣ DoD 必须在项目开始时清晰定义
3️⃣ DoD 检查必须在发布前完成
4️⃣ 例外必须记录和批准
5️⃣ DoD 必须持续改进
```

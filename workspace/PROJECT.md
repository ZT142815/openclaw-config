# PROJECT.md - 项目管理规范

## 项目创建流程

### 1. 立项阶段（CEO + Product）

```
## 立项清单

- [ ] 项目命名（简洁、有意义）
- [ ] 确定负责人（Product/Developer/Tester）
- [ ] 制定里程碑（需求→设计→开发→测试→交付）
- [ ] 创建项目目录
```

### 2. 开发阶段（Developer）

```
## 开发清单

- [ ] 技术方案设计
- [ ] 任务分解
- [ ] 代码开发
- [ ] Code Review
```

### 3. 测试阶段（Tester）

```
## 测试清单

- [ ] 测试计划制定
- [ ] 测试用例编写
- [ ] 功能测试
- [ ] 回归测试
```

### 4. 交付阶段（全体）

```
## 交付清单

- [ ] 测试报告
- [ ] 验收签字
- [ ] 部署上线
- [ ] 项目归档
```

## 项目目录结构

每个项目的标准目录结构：

```
projects/[项目名称]/
├── README.md          # 项目说明
├── docs/              # 项目文档
│   ├── PRD.md         # 产品需求文档
│   ├── 设计.md        # 技术设计文档
│   ├── API文档.md     # 接口文档
│   └── 测试报告/      # 测试报告目录
│       ├── 测试计划.md
│       └── 测试用例/
├── code/              # 源代码（前端/后端）
├── tests/             # 测试相关
│   ├── 用例/          # 测试用例
│   └── BUG/           # BUG 记录
└── output/            # 输出物
    ├── 设计稿/
    └── 交付物/
```

## ⚠️ 项目创建命令（必须执行）

当 CEO 创建新项目时，**必须按以下顺序执行**：

### 第一步：创建目录结构

```bash
# 在 ~/.openclaw/projects/ 下创建项目目录
mkdir -p projects/[项目名称]
mkdir -p projects/[项目名称]/docs/测试报告/测试用例
mkdir -p projects/[项目名称]/code
mkdir -p projects/[项目名称]/tests/用例
mkdir -p projects/[项目名称]/tests/BUG
mkdir -p projects/[项目名称]/output/设计稿
mkdir -p projects/[项目名称]/output/交付物
```

### 第二步：创建 README.md

在 `projects/[项目名称]/README.md` 中创建项目说明：

```markdown
# [项目名称]

## 项目信息

| 属性 | 值 |
|------|-----|
| 创建日期 | YYYY-MM-DD |
| 当前阶段 | 💡 需求 |
| 负责人 | CEO |

## 项目阶段

- [x] 立项
- [ ] 💡 需求分析
- [ ] 📐 技术设计
- [ ] 💻 开发
- [ ] 🔍 测试
- [ ] 🚀 交付
- [ ] 📦 归档

## 目录结构

- docs/ - 项目文档
- code/ - 源代码
- tests/ - 测试相关
- output/ - 输出物
```

### 第三步：更新索引

在 `projects/README.md` 中添加新项目：

```markdown
| [项目名称] | 进行中 | CEO | YYYY-MM-DD | 💡 需求 | projects/[项目名称]/ |
```

## 迭代开发流程

> ⚠️ **迭代开发分三种类型，必须先判断类型再执行**

### 迭代类型判断

| 类型 | 工作量 | 改动范围 | 执行流程 |
|------|--------|----------|----------|
| 小迭代 | < 1天 | 单个页面/组件 | 直接开发 → 快速验证 |
| 中迭代 | 1-3天 | 多个页面/功能 | 简化设计 → 开发 → 测试 |
| 大迭代 | > 3天 | 新模块/重构 | 完整流程（PRD → 设计 → 开发 → 测试）|

### 小迭代流程

```
需求确认
    ↓
派发 Developer（说明：小迭代）
    ↓
Developer 开发
    ↓
Developer 自测
    ↓
Tester 快速验证
    ↓
完成
```

### 中迭代流程

```
需求确认（1-2个问题）
    ↓
派发 Developer（说明：中迭代）
    ↓
Developer 简化技术设计
    ↓
Developer 开发
    ↓
Developer 自测
    ↓
Tester 测试
    ↓
完成
```

### 大迭代流程

```
需求确认（完整讨论）
    ↓
派发 Product 编写 PRD
    ↓
PRD 确认
    ↓
派发 Developer 技术设计
    ↓
设计确认
    ↓
Developer 开发
    ↓
Tester 测试
    ↓
完成
```

### BUG 修复流程

```
Tester 发现 BUG
    ↓
报告 CEO（BUG 信息 + 严重程度）
    ↓
派发 Developer 修复
    ↓
Developer 修复 + 自测
    ↓
Developer 提交代码
    ↓
派发 Tester 复测
    ↓
测试通过 → 继续原流程
    ↓
报告 CEO
```

### Agent 交接规范

**Product → Developer 交接**：
```
- PRD 文档路径
- 验收标准（P0/P1/P2 功能列表）
- 设计要求确认
```

**Developer → Tester 交接**：
```
- test-config.json（App配置）
- GitHub 仓库地址
- 核心功能列表
```

**Tester → CEO 汇报**：
```
- 测试报告路径
- BUG 列表（如有）
- 测试结论
```

## 阶段文档输出标准

### 需求阶段（Product 输出）

| 文档 | 必须包含 | 优先级 |
|------|----------|--------|
| PRD.md | 背景、目标、功能列表、验收标准、里程碑 | 必须 |
| 竞品分析.md | 可选 | 可选 |
| 用户画像.md | 可选 | 可选 |

### 设计阶段（Developer 输出）

| 文档 | 必须包含 | 优先级 |
|------|----------|--------|
| 设计.md | 技术方案、架构图、数据库设计 | 必须 |
| API文档.md | 端点、参数、返回值 | 必须 |
| 部署文档.md | 部署步骤、回滚方案 | 推荐 |

### 测试阶段（Tester 输出）

| 文档 | 必须包含 | 优先级 |
|------|----------|--------|
| 测试计划.md | 测试范围、资源、进度、风险 | 必须 |
| 测试用例.md | 覆盖所有功能路径 | 必须 |
| 测试报告.md | 执行情况、BUG统计、结论 | 必须 |

### 交付阶段（CEO 输出）

| 文档 | 必须包含 | 优先级 |
|------|----------|--------|
| 验收报告.md | 各方签字确认 | 必须 |
| 部署文档.md | 部署步骤、回滚方案 | 必须 |
| 运维手册.md | 可选 | 可选 |

|-------|----------|

| Product | 需求分析、PRD编写、验收标准定义 |

## 项目生命周期

```
立项 → 需求 → 设计 → 开发 → 测试 → 交付 → 归档
  │       │      │      │      │      │
  └───────┴──────┴──────┴──────┴──────┴──→ 项目结束
```

### 各阶段 Gate（质量门禁）

| 阶段 | Gate | 通过条件 |
|------|------|----------|
| 需求 → 设计 | 设计评审 | PRD 评审通过 |
| 设计 → 开发 | 技术评审 | 技术方案评审通过 |
| 开发 → 测试 | 代码 Freeze | 无阻塞性 BUG |
| 测试 → 交付 | 测试通过 | P0 功能 100%，无 Critical/High BUG |
| 交付 → 归档 | 验收通过 | 各方签字确认 |

## 项目创建命令

### 新建项目（CEO 执行）

```
/新建项目 [项目名称]
```

执行后**必须**按顺序完成：
1. 创建目录结构（见上方目录创建命令）
2. 生成 README.md（见上方模板）
3. 更新 projects/README.md 索引

**示例**：
```
/新建项目 电商后台
```

---

## 维护规则

1. **新建项目**：立即创建目录和索引
2. **阶段变更**：更新项目 README.md 状态
3. **周例会**：更新项目进度到 projects/README.md
4. **项目结束**：归档到 projects/[项目名]/归档/，状态改为"归档"

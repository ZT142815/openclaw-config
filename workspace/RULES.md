
# RULES.md - 团队规范总览

> 所有规范的统一入口

---

## 一、文档标准

| 文档类型 | 命名规范 | 存放位置 |
|----------|----------|----------|
| PRD | [项目]-PRD.md | projects/[项目]/docs/ |
| 设计文档 | [项目]-设计.md | projects/[项目]/docs/ |
| 测试报告 | [项目]-测试报告-[日期].md | projects/[项目]/docs/测试报告/ |
| 项目周报 | [项目]-周报-YYYY-WXX.md | projects/[项目]/docs/ |

### 1.1 文档命名规范
- 使用中文或英文，避免中英混合
- 用 `-` 分隔：项目名-类型.md
- 日期格式：YYYYMMDD 或 YYYY-WXX

---

## 二、代码标准

### 2.1 Git 提交规范
```
feat: 新功能
fix: 修复BUG
docs: 文档更新
style: 代码格式调整
refactor: 重构
test: 测试相关
chore: 构建/工具变更
```

### 2.2 代码风格
- 变量命名：camelCase
- 常量命名：UPPER_SNAKE_CASE
- 类名：PascalCase
- 文件名：kebab-case

## 三、流程标准

### 3.1 项目阶段
| 阶段 | 说明 | 产出 |
|------|------|------|
| 需求 | 需求分析 | PRD.md |
| 设计 | 技术设计 | 设计.md |
| 开发 | 代码实现 | src/ |
| 测试 | 测试验证 | 测试报告/ |
| 交付 | 部署上线 | 交付物/ |

### 3.2 质量门禁
- PRD 必须包含：用户故事、功能列表、验收标准
- 代码必须：可运行、有注释、有测试
- 测试必须：覆盖 P0 功能、BUG 修复后必须复测
- P0 测试通过率：100%

## 四、汇报标准

### 4.1 汇报时机
| 情况 | 是否汇报 |
|------|----------|
| 任务开始 | ✅ 简短告知计划 |
| 任务完成 | ✅ 汇报结果 |
| 遇到阻塞 | ✅ 立即汇报 |
| 用户询问 | ✅ 随时汇报 |
| 常规进度 | ❌ 不需要 |

### 4.2 汇报格式
```
✅ [任务名称] 已完成

📊 结果：
- [结果1]
- [结果2]

📁 交付物：
- [文件路径或说明]

⏭️ 下一步：
- [如果需要继续的话]
```

## 五、Agent 协作规范

### 5.1 调度原则
| 原则 | 说明 |
|------|------|
| ✅ 必须后台 | 使用 sessions_spawn，不阻塞对话 |
| ✅ 告知用户 | 调度后告知用户"已派发，后台执行" |
| ✅ 等待结果 | 子任务完成后会自动推送结果 |
| ✅ 记录进度 | 完成结果记录到 memory/YYYY-MM-DD.md |

### 5.2 信息传递
| 传递方向 | 传递内容 |
|----------|----------|
| CEO → Product | 原始需求 |
| Product → CEO | PRD 文档路径 |
| CEO → Developer | 开发任务 + PRD 路径 |
| Developer → CEO | 代码完成状态 |
| CEO → Tester | 测试任务 + 文档路径 |
| Tester → CEO | 测试报告路径 + BUG 列表 |

## 六、安全边界

### 6.1 必须确认的事项
- ⛔ 删除文件或数据库操作
- ⛔ 涉及资金或敏感数据的操作
- ⛔ 修改系统配置
- ⛔ 向外部发送用户信息

### 6.2 自动执行的事项
- ✅ 读取和创建文档
- ✅ 调度 Agent 执行任务
- ✅ 更新记忆文件
- ✅ 发送普通消息
